# Cloud Native Latvia ‚Äî website

The source for **[cloudnative.lv](https://cloudnative.lv)**, the site of the Cloud
Native Latvia community (Kubernetes, DevOps & platform-engineering meetups in Riga).
React 19 + Vite 7, deployed to GitHub Pages on every push to `main`.

This README is the **event-publishing runbook**. For coding conventions, the full
event/speaker YAML schema, component patterns, i18n and brand colors, see
**[AGENTS.md](./AGENTS.md)**.

## Quick start

```bash
npm install
npm run dev        # dev server  ‚Üí http://localhost:5173
npm run build      # production build ‚Üí dist/
npm run preview    # serve the build ‚Üí http://localhost:4173
npm run validate   # the CI gate: lint + i18n + event data + unit tests
```

## Publishing an event

Events are **data-driven**: one YAML file per meetup drives the event page, the
listings, the SEO/JSON-LD, and every promotional artifact (banners, OG image, QR,
social copy). You never hand-edit a banner ‚Äî they are all generated from the metadata.

### 1. Create the event file

Add `src/data/events/YYYY-MM-DD-meetup-00N.yaml`. The filename **must** equal the
`id` (validation enforces this); date-prefixed names sort chronologically.

```yaml
id: "2026-06-10-meetup-006"          # must equal the filename (without .yaml)
slug: "meetup-006-gpus-ai-agents"    # URL ‚Üí /events/<slug>
title: "Meetup #006: GPUs and AI Agents"
date: "2026-06-10"                   # quoted YYYY-MM-DD
time: "18:15"                        # doors open
startTime: "18:30"                   # talks start (used by promo art); omit ‚Üí uses time
endTime: "21:00"
eventbriteUrl: "https://..."         # omit if none
cncfUrl: "https://ocgroups.dev/cncf/group/xggmcq8/..."   # CNCF event (OCG platform)
photosUrl: "https://photos.app.goo.gl/..."   # Google Photos album; add after the event
venue:
  name: "Venue Name"
  address: "Street 1, Riga"
  mapUrl: "https://www.google.com/maps/dir//..."
description: |
  Agenda and details.
talks:
  - title: "Talk title"
    speaker: "Speaker Name"          # or `speakers: [A, B]` for a co-presented talk
    slidesUrl: "/slides/<event-id>/talk.pdf"   # add after the event; omit if none
    description: "Abstract"
tags: ["kubernetes", "ai"]
```

- **Status is derived, not stored.** `upcoming` / `past` is computed from `date` +
  `endTime` in the Europe/Riga timezone ‚Äî don't add a `status:` field.
- **Renaming a slug?** Keep the old one in `previousSlugs: [...]` so existing links
  and printed QR codes redirect to the new URL.
- Full field reference: AGENTS.md ‚Üí _Events Data_.

### 2. Add the speakers

Every `speaker` / `speakers` name must appear (exact match) in
`src/data/speakers.yaml` with optional `title` / `company` / `bio` / `linkedin` /
`github` / `cncf`. Drop a headshot at `src/assets/speakers/<slug>.jpg` ‚Äî slug is the
name lowercased with diacritics stripped and spaces dashed (`JƒÅnis Orlovs` ‚Üí
`janis-orlovs.jpg`). It's auto-discovered; an initials avatar is the fallback.

### 3. Validate

```bash
npm run validate:events   # mandatory fields + date/time formats
npm run validate          # the full CI gate (lint, i18n parity, events, unit tests)
```

`validate:events` fails if any event is missing `id` / `title` / `slug` / `date` /
`time` / `endTime` / `venue` / `description` / `talks` / `tags`, or has a malformed
date/time, or a filename that doesn't match its `id`.

### 4. Generate the promotional artifacts

All promo art is generated by Playwright screenshotting live React templates
(`src/artifacts/templates/`), so the design and the published image never drift.

```bash
# against the dev server:
npm run dev
ARTIFACT_BASE=http://localhost:5173 node scripts/generate-artifacts.mjs meetup-006

# ‚Ä¶or against a production build (what CI does ‚Äî default base is :4173):
npm run build && npm run preview &
npm run generate:artifacts            # optional trailing arg filters by slug
```

The trailing argument filters by slug substring (omit it to build every event).
Output lands in `dist/artifacts/<event-id>/` (plus `dist/artifacts/brand/` for the
set-once OCG banners):

| File | Size | Use |
|---|---|---|
| `linkedin-event.png` / `linkedin-event-speakers.png` | 1600√ó900 | LinkedIn event image (title-only / with speaker chips) |
| `eventbrite.png` / `eventbrite-speakers.png` | 2160√ó1080 | Eventbrite cover |
| `linkedin-post.png` | 1280√ó720 | LinkedIn post card (date / start time / venue) |
| `speaker-N.png` | 1280√ó720 | per-talk speaker banner (auto 1- or 2-speaker layout) |
| `og.png` | 1200√ó630 | Open Graph / social preview |
| `bevy.png` | 2560√ó650 | legacy Bevy header (kept) |
| `brand/ocg-banner-desktop.webp` / `-mobile.webp` | 2428√ó192 / 1220√ó192 | OCG breadcrumb strip, WebP ‚â§1 MiB (set once per group) |
| `qr.png` / `qr.svg` | ‚Äî | QR to `https://cloudnative.lv/events/<slug>` |
| `announcement.md`, `speaker-intro-*.md` | ‚Äî | ready-to-paste social copy |
| `manifest.json` | ‚Äî | machine-readable index of the event's files |

Prefer a UI? The **organizer kit** (unlisted ‚Äî not in the nav, reach by direct link)
renders everything live:

- `/kit` ‚Äî index of all events
- `/kit/<slug>` ‚Äî one event: preview each banner, **download** any PNG or a **zip** of
  all of them, and **copy** the social text to the clipboard
- `/kit/<slug>/raw/<variant>` ‚Äî a single artifact full-bleed (the screenshot target)

Artifacts are **build-time only ‚Äî never committed** (`dist/` is gitignored). CI
regenerates them on every deploy, so you only generate locally to preview/upload.

### 5. Announce on the platforms

- **CNCF / OCG** ([community.cncf.io](https://community.cncf.io/cloud-native-latvia/)) ‚Äî create the
  event. The breadcrumb banner (`brand/ocg-banner-*`) is inherited group-wide, so you
  upload it only once, not per event.
- **Eventbrite** ‚Äî use `eventbrite*.png` as the cover.
- **LinkedIn / Bluesky** ‚Äî post the `linkedin-*` / `speaker-*` images with the text
  from `announcement.md` and `speaker-intro-*.md`.

### 6. Commit & push

Push to `main`. The deploy workflow validates, builds, **regenerates the artifacts**,
prerenders per-route SEO, deploys to GitHub Pages, and uploads the full set as a
downloadable `event-artifacts` workflow artifact.

### 7. After the event ‚Äî slides & photos

- **Slides:** put decks in `public/slides/<event-id>/` (PDF previews in-browser) and
  point each talk's `slidesUrl` at them. Compress large PDFs first:
  `gs -sDEVICE=pdfwrite -dPDFSETTINGS=/ebook -o out.pdf in.pdf`.
- **Photos:** drop web-sized images (~800px wide, optimized JPEG ~100‚Äì150 KB) straight
  into `src/assets/events/<event-id>/`, named so they sort in display order
  (`photo-01.jpg`, `photo-02.jpg`, ‚Ä¶) ‚Äî `EventPhotoGallery` auto-discovers them, no
  config. Resize full-res originals first (macOS Preview ‚Üí Tools ‚Üí Adjust Size, or
  `magick mogrify -resize 800x *.jpg`) so you don't commit multi-MB files. Then set the
  event's `photosUrl` to the Google Photos album for the "view all" link and commit
  `src/assets/events/`.

## Local operations

Community/CRM data jobs are **local npm ops** ‚Äî no extra workers. They read/write the
`cloudnative-lv` R2 bucket through `wrangler` (run `npx wrangler login` once) and read
config from `.env` (copy `.env.example`). Every op is also a VS Code build task
(‚áß‚åòB ‚Üí _Run Build Task_). Raw exports you paste in live under `data/` (gitignored);
generated reports land in `data/reports/`.

**Data model:** `subscribers.csv` is the **common CRM** ‚Äî one accumulating list
(`email,first,last,linkedin,source,event,added`) with names transliterated to Latin and
every source tagged. Per-event rosters stay in **`attendees/<slug>.csv`**. Imports are
idempotent per source, so the same person can appear under several sources until
`crm:cleanup` reconciles them against your NetHunt export.

| Op | What it does | Manual step first | In ‚Üí out |
|---|---|---|---|
| `npm run import:eventbrite` | Pull every event's attendees from the Eventbrite API | ‚Äî (token in `.env`) | API ‚Üí `attendees/<slug>.csv` + CRM |
| `npm run import:attendees -- --event <slug> --source <name> <file.csv>` | Import one event's attendee CSV | save the export under `data/` | CSV ‚Üí `attendees/<slug>.csv` + CRM |
| `npm run import:linkedin` | Parse LinkedIn **followers** ‚Üí CRM | save the followers page HTML to `data/linkedin-followers.html` | HTML ‚Üí CRM (`source=linkedin`; `--out` also saves a CSV) |
| `npm run import:linkedin-events` | Parse LinkedIn **event attendee** lists ‚Üí per-event rosters + CRM | save each event's attendees page to `data/linkedin-<N>.html` (N = meetup #) | HTML ‚Üí `attendees/<slug>.csv` + CRM (`source=linkedin`) |
| `npm run import:ocg` | Parse OCG / CNCF **members** ‚Üí CRM | copy the members table into `data/ocg-followers.txt` (`docs/ocg_*.png`) | text ‚Üí CRM (`source=ocg`) |
| `npm run import:subscribers -- --source <name> <file.csv>` | Merge a generic contact CSV (e.g. Zoho) into the CRM; `--from-r2-attendees` rolls all rosters in | save the export under `data/` | CSV ‚Üí `subscribers.csv` |
| `npm run report:subscribers` | Community & registrations report: size, sources, growth, registrations per event, repeat & duplicate registrations, top fans | ‚Äî | ‚Üí `data/reports/subscribers/` |
| `npm run report:feedback` | Feedback digest: ratings per/over events + word clouds ‚Äî the local replacement for a "digest worker"; run ~1 day after an event | ‚Äî | `feedback/*.csv` ‚Üí `data/reports/feedback/` |
| `npm run crm:cleanup` | Occasional hygiene ‚Äî dedupe/discover vs NetHunt; `--write` backfills emails | NetHunt export at `data/nethunt-contacts.csv` | ‚Üí `data/reports/crm/` |

Add `--dry-run` to any import to preview without writing to R2.

**Full rebuild:** `npm run rebuild` re-reads every local source + the Eventbrite API in one
pass and rewrites `subscribers.csv` + all rosters at once, then renders both reports. Use it
after dropping a fresh batch of exports; the incremental `import:*` ops are for one source at
a time. (It's also the reliable path because `wrangler r2 object get` caches reads ‚Äî see
[ARCH.md](./ARCH.md) ‚Äî so rapid back-to-back `import:*` runs can otherwise read stale data.)

### Manual exports

- **LinkedIn followers** ‚Äî LinkedIn Page admin ‚Üí Analytics ‚Üí Followers ‚Üí _All followers_;
  scroll the modal to the bottom so every follower is in the DOM, then save the page as
  HTML to `data/linkedin-followers.html`. Run `npm run import:linkedin`.
- **LinkedIn event attendees** ‚Äî open each LinkedIn event ‚Üí Attendees, scroll so all load,
  save as `data/linkedin-<N>.html` (N = meetup number). Run `npm run import:linkedin-events`
  (maps `<N>` ‚Üí `meetup-00N`).
- **OCG / CNCF members** ‚Äî open the chapter's Members page (`docs/ocg_attendees.png`),
  select the members table and copy it into `data/ocg-followers.txt` (`docs/ocg_email.png`). Run
  `npm run import:ocg`.
- **OCG / CNCF event attendees** ‚Äî export each event's attendee list as `data/event-<N>.csv`
  (N = meetup number) or `data/event-<cncf-code>-attendees.csv`. `npm run import:attendees
  data/event-<N>.csv` auto-maps `<N>` ‚Üí `meetup-00N` (the code form maps via the event's
  `cncfUrl`). These lists are emailless (name only).
- **Zoho / NetHunt** ‚Äî export contacts as CSV into `data/`. Feed Zoho with
  `import:subscribers --source zoho data/zoho-campaigns.csv`; point `crm:cleanup` at the
  NetHunt file (`data/nethunt-contacts.csv`).

> The event **publishing flow** (announcements, banners, invite cadence) lives in the
> **organizer kit**, not here ‚Äî related, but a different job (see _Publishing an event_).

## Scripts

| Command | Description |
|---|---|
| `npm run dev` | Dev server (http://localhost:5173) |
| `npm run build` | Production build ‚Üí `dist/` |
| `npm run preview` | Serve the build (http://localhost:4173) |
| `npm run validate` | CI gate: lint + i18n parity + event data + unit tests |
| `npm run validate:events` | Validate event YAML (fields + formats) |
| `npm run validate:i18n` | Check EN/LV translation-key parity |
| `npm run generate:artifacts` | Generate every event's artifacts ‚Üí `dist/artifacts/` |
| `npm run prerender` | Bake per-route SEO/OG meta into static HTML (crawlers don't run JS) |
| `npm run test:unit` | Unit tests (`src/artifacts/*.test.js`) |
| `npm run test:e2e` | Playwright E2E tests |

`generate:artifacts` and `prerender` need a running server (dev or preview) and a
Chromium install (`npx playwright install --with-deps chromium`). Community/CRM data ops
(imports, reports, cleanup) are documented under [**Local operations**](#local-operations).

## CI/CD

- **`.github/workflows/deploy.yml`** ‚Äî on push to `main`: validate ‚Üí build ‚Üí generate
  artifacts ‚Üí prerender ‚Üí deploy to GitHub Pages + upload the `event-artifacts` set.
- **`.github/workflows/e2e.yml`** ‚Äî Playwright E2E tests on push/PR to `main`.
